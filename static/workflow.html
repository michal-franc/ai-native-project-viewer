Agent Operating Flow

Step-by-step sequence an agent follows when picking up and completing work.

1
Learn the project
Run issue-cli process to understand the workflow, rules, and conventions. Do this once at the start.
2
Find available work
Run issue-cli next --version 0.1 to see backlog, in-progress, and testing issues. Use --design for ideas needing design work. Use issue-cli search <query> to find issues by title, body, or status.
3
Start work with issue-cli start <slug>
This is the critical step. start does three things: claims the issue, transitions it to in progress, and shows next steps with checklist items. Do NOT use claim to begin work — it only sets the assignee without transitioning.
4
Do the work, check off items
Implement the feature or fix. Check off [x] checkboxes in the issue body as acceptance criteria are met. All checkboxes must be checked before moving to testing.
5
Add test plan
Add a ## Test Plan section to the issue body with ### Automated and ### Manual subsections. Then transition: issue-cli transition <slug> --to "testing"
6
Log test results
Run tests and log results: issue-cli comment <slug> --text "tests: 5 unit tests, all passing". Then transition to documentation: issue-cli transition <slug> --to "documentation"
7
Update documentation
Update any relevant docs, then log it: issue-cli comment <slug> --text "docs: updated guides/workflow.md"
8
Mark done
Run issue-cli done <slug>. This validates all requirements across remaining statuses, transitions to done, and auto-unclaims the assignee.

Transition Gates

Each transition enforces validation rules. The agent cannot skip steps.

TransitionRequirementWhy
idea → in design body_not_empty Must describe the idea before designing it
in design → backlog has_checkboxes Acceptance criteria must be defined as checkboxes
backlog → in progress has_assignee Someone must own the work — use start
in progress → testing all_checkboxes_checked All acceptance criteria must be met before testing
testing → documentation has_test_plan comment: tests: Test plan with automated + manual sections, plus logged results
documentation → done all_checkboxes_checked comment: docs: Docs must be updated and logged before shipping

Command Reference

Core workflow (use these)

issue-cli start <slug>
Begin work. Claims the issue, transitions to in-progress, shows checklist and next steps.
This is how agents should always start work.
issue-cli next --version <v>
Find available work for a version. Shows backlog (unclaimed), in-progress, and testing issues.
issue-cli transition <slug> --to "<status>"
Move issue to next status. Validates requirements and appends status templates.
issue-cli done <slug>
Mark complete. Validates all remaining gates, transitions to done, auto-unclaims.

Information & search

issue-cli process
Learn the project workflow, rules, and conventions. Run this first.
issue-cli context <slug>
Full dump: body, checklist, test plan status, comments.
issue-cli search <query>
Search across titles, bodies, and statuses.
issue-cli list --status/--system/--assignee
Filtered list. Shows "claimed by X" for assigned issues.
issue-cli stats
Project health: issue counts by status, system, assignee.
issue-cli checklist <slug>
Show checkbox progress for an issue.

Updates

issue-cli create --title "..." --system X
Create a new issue. Defaults to "idea" status. Applies workflow templates.
issue-cli comment <slug> --text "..."
Add a comment. Use "tests:" and "docs:" prefixes for validation gates.
issue-cli claim <slug> --assignee "..."
Only sets assignee. Does NOT transition status.
Do not use this to begin work. Use 'start' instead.
issue-cli unclaim <slug>
Remove assignee from an issue.

Key Rules for Agents

▶
Use start, not claim
start claims AND transitions to in-progress. claim only sets the assignee — the issue stays in backlog.
→
Never skip statuses
Transitions are strictly sequential. The CLI enforces this — you cannot jump from backlog to testing.
☐
Check off acceptance criteria
All [x] checkboxes must be checked before transitioning from in-progress to testing.
📝
Test plan required
Add ## Test Plan with ### Automated and ### Manual subsections before leaving testing.
💬
Log with comment prefixes
Use tests: prefix after testing and docs: prefix after updating documentation. These are validated.
✅
done validates everything
issue-cli done checks all remaining gates at once and auto-unclaims the assignee on success.
