Agent Execution Flow

From dispatch prompt to approvals, docs, retrospectives, and bug reports.

The page below shows the intended top-to-bottom path an agent follows in this app: how it starts, how workflow guidance is layered, what each stage means, when it must stop, and where the side effects are stored.

START
Dispatch

Server builds the base prompt

The app dispatches an issue with generic operational instructions plus current-status workflow guidance.

  • Runs through `issue-cli process workflow` and `issue-cli process transitions` first.
  • Includes a generic goal instead of hardcoded stop statuses.
  • Injects the current `status.prompt` for the issue's status.
  • Tells the agent how to report bugs and save retrospectives.
Dispatch also exports `ISSUE_CLI_LOG` and `ISSUE_VIEWER_SERVER_PWD` into the tmux session.
IDEA
Status

Clarify with the human first

The agent should explore the idea with the user before prematurely designing or implementing.

  • Ask focused questions about scope, goals, and success criteria.
  • Identify affected systems and player impact.
  • Capture ambiguity early instead of guessing.
DESIGN
Status + Transition

Enter structured design work

Moving into design creates the scaffolding for design and human questions. The status prompt then drives research and planning behavior.

  • `idea -> in design` validates non-empty issue body.
  • Appends `Idea`, `Design`, and `Human Input` sections.
  • Agent reviews docs and code before proposing concrete changes.
  • Open questions and required human input are made explicit.
READY
Approval Gate

Backlog means designed and approved

Leaving design is not just a checklist event. It is where the workflow enforces readiness and human approval.

  • `Idea`, `Design`, and `Human Input` must be complete.
  • Human approval is required before the issue becomes backlog.
  • The transition appends `Readiness` and clears the assignee.
`Readiness` is intentionally reviewed on the next transition by whoever picks the work up.
BUILD
Implementation

Backlog to active work

Implementation begins only after an assignee exists and the readiness checklist has been completed.

  • `backlog -> in progress` requires an assignee.
  • Validates the `Readiness` section.
  • Appends `Implementation` and `Test Plan` sections.
  • The status prompt keeps the agent aligned with the accepted design.
TEST
Verification

Build automated coverage and test evidence

Testing is where implementation completeness turns into verification completeness.

  • `in progress -> testing` validates `Implementation` and `Test Plan`.
  • Appends the `Testing` section.
  • Status guidance emphasizes relevant unit tests and E2E tests.
  • Subsystem overlays can add deeper testing expectations for systems like `Combat`, `UI`, or `Data`.
HUMAN
Manual Gate

Prepare and wait for human verification

The bot can prepare manual verification, but it should not pretend to complete it.

  • `testing -> human-testing` validates `Testing` and requires a `tests:` comment.
  • Appends `Human Testing` with explicit, reproducible checks.
  • `human-testing -> documentation` requires `Human Testing`, `tests:`, and human approval.
DOCS
Documentation

Update the right docs for the system

Documentation is deliberately after human verification. At this point the bot should update the docs that match the affected subsystem.

  • Subsystem overlays point the agent at concrete doc areas such as `docs/combat/`, `docs/ui/`, or `docs/data/`.
  • The bot records doc work with a `docs:` comment.
  • `documentation -> done` stays mechanical once docs are complete.
DONE
Complete

Ship, document, and stop

When the issue reaches `done`, the workflow has already enforced implementation, testing, human validation, and documentation.

First commands

issue-cli process workflow issue-cli process transitions issue-cli start <slug> issue-cli show <slug>

For backlog issues, start only works after a human approves in progress in the issue viewer.

How guidance is layered

Two workflow layers are active at runtime.

  • Status prompt: how to behave while in the current status.
  • Transition actions: validations, append rules, injected prompts, field updates, and approvals for the exact move.

When the bot must stop

If any of these are required, the bot should stop instead of guessing.

Human approval Manual verification User clarification Workflow blockage Tooling blockage

Retrospective command

When the bot stops for one of those reasons, it should save workflow feedback into the project.

issue-cli retrospective <slug> --body "Base workflow: ... Subsystem workflow for <System>: ... Missing system-specific instructions: ... Tooling/workflow friction: ..."

This writes a markdown file under the project `retros/` directory.

Bug report command

If the problem is in `issue-cli` itself, the bot should file a tool bug instead of burying it in the retrospective.

issue-cli report-bug "description of what went wrong"

This writes JSON into `bugs/` under the server root for dispatched sessions, or under the current shell working directory for manual runs.

Persistence map

  • `workflow.yaml` stores workflow semantics.
  • Browser `localStorage` stores workflow-designer draft layout.
  • Project `retros/` stores workflow retrospectives.
  • `bugs/` stores `issue-cli` bug reports.
  • Issue markdown files store issue state, comments, and body updates.
